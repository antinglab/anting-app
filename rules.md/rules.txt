# 앤팅(Anting) 프로젝트 규칙

> 이 파일은 프로젝트 최상단에 위치하며, 모든 개발 작업의 기준이 됩니다.
안티그래비티(Antigravity) 에이전트는 작업 시작 전 반드시 이 파일을 먼저 읽고 모든 규칙을 준수해야 합니다.
> 

---

## Section 1 — 기술 스택 (Tech Stack)

### 프론트엔드

- **Framework**: Next.js 14 (App Router)
- **Platform**: Firebase Hosting
- **Language**: TypeScript (strict 모드, `any` 사용 금지)
- **Styling**: Tailwind CSS + CSS Variables
- **폰트**: Pretendard (CDN), DM Serif Display (Google Fonts)
- **아이콘**: Lucide React
- **에디터**: TipTap (리치 텍스트)
- **폼 검증**: React Hook Form + Zod
- **차트**: Recharts
- **엑셀 처리**: SheetJS (xlsx)

### 백엔드

- **Platform**: Firebase (전체 백엔드)
- **Database**: Firebase Firestore
- **Authentication**: Firebase Auth (카카오·네이버 소셜 로그인)
- **Storage**: Firebase Storage
- **Functions**: Firebase Cloud Functions (Node.js 20)
- **Scheduler**: Firebase Cloud Scheduler
- **Hosting**: Firebase Hosting

### AI / 외부 API

- **텍스트 생성**: Gemini Pro API (`@google/generative-ai`)
- **이미지 생성**: Vertex AI Imagen API
- **영상 생성**: Creatomate API
- **검색**: Algolia
- **결제**: 토스페이먼츠 SDK
- **알림**: 카카오 알림톡 + Firebase Cloud Messaging (FCM)
- **배송**: CJ대한통운 / 롯데택배 API
- **이메일**: SendGrid

### 개발 도구

- **에디터**: Google Antigravity 2.0
- **버전 관리**: Git + GitHub
- **배포**: Firebase CLI (`firebase deploy`)
- **패키지 매니저**: npm
- **환경 변수**: `.env.local` (로컬), Firebase Functions Config (서버)

## Section 2 — 개발 규칙 (Development Rules)

### 2-1. 작업 완료 시 필수 절차 (매 작업마다 반드시 이행)

`① 기능 구현 완료
② 오류 검토 (TypeScript 타입 오류, 콘솔 에러, 빌드 오류 전수 확인)
③ 문제 없음 확인 후 → Firebase 에뮬레이터 또는 스테이징에서 동작 검증
④ 작업 결과를 .md 파일로 저장 (파일명: YYYY-MM-DD_작업명.md)
⑤ 한국어로 완료 보고`

### 2-2. 작업 완료 보고 형식 (.md 파일 + 보고 메시지)

작업이 완료될 때마다 아래 형식으로 `.md` 파일을 생성하고 동일 내용을 보고합니다.

markdown

`# 작업 완료 보고

## 작업명
[구현한 기능명]

## 작업 일시
YYYY-MM-DD HH:MM

## 구현 내용
- [완료된 항목 1]
- [완료된 항목 2]

## 생성/수정된 파일
- `경로/파일명.tsx` — 설명
- `경로/파일명.ts` — 설명

## 오류 검토 결과
- TypeScript 오류: 없음 / [있으면 목록]
- 빌드 오류: 없음 / [있으면 목록]
- 콘솔 에러: 없음 / [있으면 목록]

## 동작 확인
- [ ] 로컬 에뮬레이터에서 정상 동작 확인
- [ ] 모바일 뷰 정상 확인

## 다음 작업
[다음에 진행할 작업]`

### 2-3. 오류 처리 원칙

- TypeScript `strict` 모드 — `any` 타입 사용 절대 금지, 올바른 타입 정의 필수
- 빌드 오류가 있으면 해결 후 보고 (오류가 있는 상태로 완료 선언 금지)
- API 호출은 반드시 `try/catch`로 감싸고 사용자에게 에러 메시지 표시
- Firebase Security Rules 변경 시 반드시 명시적으로 보고

### 2-4. 커밋 & 배포 규칙

- 커밋 메시지 형식: `feat: 기능명`, `fix: 수정내용`, `refactor: 리팩토링 내용`
- 기능 단위로 커밋 (하나의 커밋에 여러 기능 혼재 금지)
- 배포 명령어: `firebase deploy` (전체) 또는 `firebase deploy --only hosting` (프론트만)
- 프로덕션 배포 전 반드시 `npm run build` 성공 확인

### 2-5. 보고 언어 규칙

- **모든 보고, 설명, 질문은 한국어로** 작성합니다
- 코드 주석: 한국어 또는 영어 모두 허용
- 변수명·함수명: 영어 (camelCase)
- 파일명: 영어 (kebab-case) 또는 PascalCase (컴포넌트)
